2019

Project 1

Task 1 CLOCK replacement policy

Preliminary

Project #1 - Buffer Pool | CMU 15-445/645 :: Intro to Database Systems (Fall 2019)

clock replacement policy近似LRU替换算法。
具体地，对于该算法而言，需要一下成员：

  • 记录clock中的帧
  • 一个时针指向某一帧

可以参考项目介绍。

以下是需要我们实现的函数：

  1. 需要实现的子类是ClockReplacer：

    • 头文件：src/include/buffer/clock_replacer.h
    • 实现文件：src/buffer/clock_replacer.cpp

  2. You will need to implement the clock policy discussed in the class. You will need to implement the following methods:

    • Victim(T*) :
      • Starting from the current position of clock hand, find the first frame that is both in the ClockReplacer and with its ref flag set to false.
      • If a frame is in the ClockReplacer, but its ref flag is set to true, change it to false instead.
    • Pin(T) :
      • This method should be called after a page is pinned to a frame in the BufferPoolManager.
      • It should remove the frame containing the pinned page from the ClockReplacer.
    • Unpin(T) :
      • This method should be called when the pin_count of a page becomes 0.
      • This method should add the frame containing the unpinned page to the ClockReplacer.
    • Size() :
      • This method returns the number of frames that are currently in the ClockReplacer.

  3. 需要保证线程安全。

Implementation

  1. 需要一个互斥锁来实现函数，保证函数是线程安全的。
    具体地：

    • 使用了C++提供的std::shared_mutex作为互斥锁。理由是相比于普通的std::mutex更加高效。
      • c++ - difference between std::mutex and std::shared_mutex - Stack Overflow
      • std::shared_mutex - cppreference.com
    • 使用std::lock_guard实现对于互斥锁的RAII。

  2. 确定实现clock使用的数据结构：

    • 一开始本来向利用链表实现，后面发现比较傻，因为不方便管理所有帧（在clock中和不在clock中）。
    • 参考了CMU15445 (Fall 2019) 之 Project#1 - Buffer Pool 详解 - 之一Yo - 博客园 (cnblogs.com)：
      • 利用std:vecotor作为容器存帧，
        • 每一帧使用标志记录是否在clock中，以及其ref是否为true。

  3. 实现Size() :

    1. 互斥锁。
    2. 直接遍历计数所有的in_clock位为真的数量。

  4. 实现Victim(T*)：

    1. 首先需要确定clock中帧的数量：Size() == 0直接返回；否则继续处理。
    2. 因为需要访问clock帧，所以需要加锁。
    3. 循环查找第一个在clock中的ref位为false的帧；遇到ref == true的帧，则ref = false。
    4. 找到符合条件的帧之后，将其从clock中移除，in_clock置为false；并将帧号记录到frame_id指针指向的内存位置，*frame_id = clock_cursor_。返回true。

  5. 实现Pin(T) :

    1. 判断给定的frame_id是否越界。利用assert断言进行判断。
    2. 互斥锁。
    3. 将对应的in_clock位置为false。

  6. 实现Unpin(T) :

    1. 判断给定的frame_id是否越界。利用assert断言进行判断。
    2. 互斥锁。
    3. 将对应的in_clock位置为true，ref位置为true表示刚被访问过之后加入clock。

Task2 - Buffer Pool Manager

需要实现的函数包括：

  • FetchPageImpl(page_id)
    • 根据page id，从buffer pool中获取对应的页，如果不在buffer pool中，则从磁盘读入。
  • NewPageImpl(page_id)
    • 文件系统中新建一页，并放入buffer pool中方便读写（可能涉及页替换），返回新页在内存中的指针。
  • UnpinPageImpl(page_id, is_dirty)
    • 根据页号，将对应的页面在buffer pool中的pin引用值减一，如果pin引用值为0，加入replacer。
  • FlushPageImpl(page_id)
    • 将buffer pool中的目标页写回磁盘（当页面存在buffer pool中并且脏位为真才写回），写回之后重置脏位。
  • DeletePageImpl(page_id)
    • 从缓冲池和数据库文件中删除一页。
  • FlushAllPagesImpl()
    • 将buffer pool中的所有页都写回磁盘。

每个函数都有相应的实现说明，因此难度不大。

Project 0 C++ Primer

项目任务

有两个任务需要完成：

  1. 实现字典树
  2. 用读写锁实现字典树的多线程安全访问

项目记录

  1. 首先需要切换到项目发布的时候的检查点。

    • ```bash
      git checkout 7307f94617eda60c175c6e113288e16953536f96

  1. TrieNode和TrieNodeWithValue可以根据项目的注释介绍完成。

    • 完成的过程中及时就该注释，方便自己之后实现其他函数的时候查看。

  2. 难点在于实现Trie类的三个函数，并且之后需要进一步加读写锁，实现线程安全。

    1. 首先实现单线程版的：
      1. Insert()函数：
        • 需要实现不断地往叶子节点遍历，或者节点不存在不断加入新的key char节点。
        • 需要注意区分最后一个key char的处理，分为三种情况：
          • 节点存在并且是值节点：返回false。
          • 节点存在但是普通节点：用该普通节点调用值节点的移动构造函数，替换成值节点。
          • 节点不存在：插入值节点即可
        • 新建节点的时候，尽量使用std::make_unique<>()，不要使用new。
      2. Remove()函数：
        • 删除key的时候，需要考虑两种情况：
          • 被删除的key就是一条单独的从根节点到叶子节点的路径：删除叶子节点到从低往上第一个分支的位置就行。
          • 被删除的key不只是一个值节点，它还有孩子，即它还拥有到其他key的路径：不能删除节点（理想情况是将该值节点替换成一个普通节点），但是好像不需要这么做也可以过。
          • 由于删除的过程是逆序的，因此需要使用一个栈来存储遍历的路径，最后逆序查看，确定是否删除，遇到第一个不需要删除的位置，删除结束。
      3. GetValue()函数：
        • 就是一个不断查找的过程，一路从根节点根据key char 进行查找，找到了节点之后需要进一步确定是否是值节点：
          • 使用dynamic_cast<>()将尝试将一个节点转化为一个值节点。
            • 即尝试从父类TrieNode *对象转化为一个TrieNodeWithValue *对象。
          • 如果成功了就说明找到了，否则失败。
    2. 实现线程安全，利用bustub提供的ReadWriteLatch锁进行实现（也是类中自带的）：
      1. Insert()函数：
        • 在插入的过程中可能会设计新节点的插入，因此需要的是在进行尝试插入的一开始就获取写锁latch_.WLock()；
        • 在所有可能退出的地方释放锁，防止死锁。
      2. Remove()函数：
        • 与Insert()函数类似，在删除的过程中可能发生节点的删除，并且需要保证遍历的过程中，不能有其他的线程尝试修改字典树，因此需要一开始就获取写锁。
        • 在可能退出的地方释放锁，防止死锁。
      3. GetValue()函数：
        • 与前面两个函数不同的是，GetValue()函数只需要读取信息，因此在一开始尝试查找key的时候就获取读锁latch_.RLock()。
        • 查找到节点，并且获取到值之后，就可以释放锁。
