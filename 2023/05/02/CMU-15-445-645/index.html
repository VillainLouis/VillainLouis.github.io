Project 1

Task 1 CLOCK replacement policy

Preliminary

Project #1 - Buffer Pool | CMU 15-445/645 :: Intro to Database Systems (Fall 2019)

clock replacement policy近似LRU替换算法。
具体地，对于该算法而言，需要一下成员：

  • 记录clock中的帧
  • 一个时针指向某一帧

可以参考项目介绍。

以下是需要我们实现的函数：

  1. 需要实现的子类是ClockReplacer：

    • 头文件：src/include/buffer/clock_replacer.h
    • 实现文件：src/buffer/clock_replacer.cpp

  2. You will need to implement the clock policy discussed in the class. You will need to implement the following methods:

    • Victim(T*) :
      • Starting from the current position of clock hand, find the first frame that is both in the ClockReplacer and with its ref flag set to false.
      • If a frame is in the ClockReplacer, but its ref flag is set to true, change it to false instead.
    • Pin(T) :
      • This method should be called after a page is pinned to a frame in the BufferPoolManager.
      • It should remove the frame containing the pinned page from the ClockReplacer.
    • Unpin(T) :
      • This method should be called when the pin_count of a page becomes 0.
      • This method should add the frame containing the unpinned page to the ClockReplacer.
    • Size() :
      • This method returns the number of frames that are currently in the ClockReplacer.

  3. 需要保证线程安全。

Implementation

  1. 需要一个互斥锁来实现函数，保证函数是线程安全的。
    具体地：

    • 使用了C++提供的std::shared_mutex作为互斥锁。理由是相比于普通的std::mutex更加高效。
      • c++ - difference between std::mutex and std::shared_mutex - Stack Overflow
      • std::shared_mutex - cppreference.com
    • 使用std::lock_guard实现对于互斥锁的RAII。

  2. 确定实现clock使用的数据结构：

    • 一开始本来向利用链表实现，后面发现比较傻，因为不方便管理所有帧（在clock中和不在clock中）。
    • 参考了CMU15445 (Fall 2019) 之 Project#1 - Buffer Pool 详解 - 之一Yo - 博客园 (cnblogs.com)：
      • 利用std:vecotor作为容器存帧，
        • 每一帧使用标志记录是否在clock中，以及其ref是否为true。

  3. 实现Size() :

    1. 互斥锁。
    2. 直接遍历计数所有的in_clock位为真的数量。

  4. 实现Victim(T*)：

    1. 首先需要确定clock中帧的数量：Size() == 0直接返回；否则继续处理。
    2. 因为需要访问clock帧，所以需要加锁。
    3. 循环查找第一个在clock中的ref位为false的帧；遇到ref == true的帧，则ref = false。
    4. 找到符合条件的帧之后，将其从clock中移除，in_clock置为false；并将帧号记录到frame_id指针指向的内存位置，*frame_id = clock_cursor_。返回true。

  5. 实现Pin(T) :

    1. 判断给定的frame_id是否越界。利用assert断言进行判断。
    2. 互斥锁。
    3. 将对应的in_clock位置为false。

  6. 实现Unpin(T) :

    1. 判断给定的frame_id是否越界。利用assert断言进行判断。
    2. 互斥锁。
    3. 将对应的in_clock位置为true，ref位置为true表示刚被访问过之后加入clock。

Task2 - Buffer Pool Manager

需要实现的函数包括：

  • FetchPageImpl(page_id)
    • 根据page id，从buffer pool中获取对应的页，如果不在buffer pool中，则从磁盘读入。
  • NewPageImpl(page_id)
    • 文件系统中新建一页，并放入buffer pool中方便读写（可能涉及页替换），返回新页在内存中的指针。
  • UnpinPageImpl(page_id, is_dirty)
    • 根据页号，将对应的页面在buffer pool中的pin引用值减一，如果pin引用值为0，加入replacer。
  • FlushPageImpl(page_id)
    • 将buffer pool中的目标页写回磁盘（当页面存在buffer pool中并且脏位为真才写回），写回之后重置脏位。
  • DeletePageImpl(page_id)
    • 从缓冲池和数据库文件中删除一页。
  • FlushAllPagesImpl()
    • 将buffer pool中的所有页都写回磁盘。

每个函数都有相应的实现说明，因此难度不大。

          • 被删除的key不只是一个值节点，它还有孩子，即它还拥有到其他key的路径：不能删除节点（理想情况是将该值节点替换成一个普通节点），但是好像不需要这么做也可以过。

          • 使用dynamic_cast<>()将尝试将一个节点转化为一个值节点。
            • 即尝试从父类TrieNode *对象转化为一个TrieNodeWithValue *对象。
          • 如果成功了就说明找到了，否则失败。

      1. Insert()函数：
        • 在插入的过程中可能会设计新节点的插入，因此需要的是在进行尝试插入的一开始就获取写锁latch_.WLock()；
        • 在所有可能退出的地方释放锁，防止死锁。
      2. Remove()函数：
        • 与Insert()函数类似，在删除的过程中可能发生节点的删除，并且需要保证遍历的过程中，不能有其他的线程尝试修改字典树，因此需要一开始就获取写锁。
        • 在可能退出的地方释放锁，防止死锁。
      3. GetValue()函数：
        • 与前面两个函数不同的是，GetValue()函数只需要读取信息，因此在一开始尝试查找key的时候就获取读锁latch_.RLock()。
        • 查找到节点，并且获取到值之后，就可以释放锁。

Task 2 LRU-K Replacement Policy

准备

需要实现LRUKReplacer

涉及到的文件：

  • src/include/buffer/lru_k_replacer.h
  • src/buffer/lru_k_replacer.cpp

LRU-K算法：

  • The LRU-K algorithm evicts a frame whose backward k-distance is maximum of all frames in the replacer.
  • Backward k-distance is computed as the difference in time between current timestamp and the timestamp of kth previous access.
  • A frame with less than k historical accesses is given +inf as its backward k-distance.
  • When multipe frames have +inf backward k-distance, the replacer evicts the frame with the earliest timestamp.

LRUReplacer的最大容量与buffer pool的容量相同。

  • It contains placeholders for all of the frames in the BufferPoolManager.
    • However, at any given moment not all the frames in the replacer are considered to be evictable.
  • The size of LRUKReplacer is represented by the number of evictable frames.
    • The LRUKReplacer is initialized to have no frames in it.
    • Then, only when a frame is marked as evictable, replacer’s size will increase.

需要实现的函数：

  • Evict(frame_id_t*) : Evict the frame with largest backward k-distance compared to all other evictable frames being tracked by the Replacer. Store the frame id in the output parameter and return True. If there are no evictable frames return False.
  • RecordAccess(frame_id_t) : Record that given frame id is accessed at current timestamp. This method should be called after a page is pinned in the BufferPoolManager.
  • Remove(frame_id_t) : Clear all access history associated with a frame. This method should be called only when a page is deleted in the BufferPoolManager.
  • SetEvictable(frame_id_t, bool set_evictable) : This method controls whether a frame is evictable or not. It also controls LRUKReplacer‘s size. You’ll know when to call this function when you implement the BufferPoolManager. To be specific, when pin count of a page reaches 0, its corresponding frame is marked evictable and replacer’s size is incremented.
  • Size() : This method returns the number of evictable frames that are currently in the LRUKReplacer.

具体注意事项

理解LRU-K算法

此任务的关键在于理清楚LRU-K算法的具体流程。

首先需要知道的是LRU-K replacer本质上只是buffer pool中的所有页面访问记录的一个追踪器。

具体的，k作为一个关键的参数，在replacer中选择替换页面的时候：

  • 优先从访问次数< k次的页面中寻找替换页面，并且实行的时候FIFO策略，即某一页被访问的次数不足k的时候，不采取LRU策略，而是朴素的FIFO策略；
  • 如果所有的页面访问次数都> k次，则实行LRU策略，即替换页面选择最近没有被访问的。
数据结构

由于需要对不同访问次数的页面实行不同的替换策略，因此维护两个list:

  • 访问次数< k:
    • 用一个list实现队列操作，每次在队头插入，队尾删除；
    • 记为history_。
  • 访问次数> k:
    • 用一个list实现LRU操作，访问的时候及时更新。
    • 记为buffer_。

使用两个哈希表，分别来加速history_和buffer_中的元素的访问和删除操作。

对于页面还需要设置其是否可以被替换，而页面的数量是固定的，因此使用一个向量std::vector<bool>存储页面是否可以被替换。

同样的，页面的访问次数也需要记录，并且数量是固定的，使用一个向量std::vector<size_t>存储当前页面被访问的次数。

  • 访问次数可以用来确定采用什么策略：
    • 访问次数= 1，加入history_。
    • 访问次数< k，在history_中。
    • 访问次数>= k，在buffer_中。

需要维护当前replacer的大小cur_size_，即replacer中可以被替换的页面总数。
